## 1. Общая подробная концепция

### 1.1 Назначение продукта
K.O.T (King Of Time) — приложение для организации мероприятий с фокусом на согласование времени, участие людей и управляемую коммуникацию внутри события.

Приложение решает задачи:
- Создание и управление мероприятиями.
- Набор участников и обработка заявок.
- Голосование за слоты времени.
- Коммуникация участников в интегрированном чате.
- Управление расходами по событию.
- Персонализация опыта (язык, тема, профиль, премиум).

### 1.2 Продуктовые принципы
1. **MVP-first**: минимально достаточная функциональность без дорогих инфраструктурных решений.
2. **Прозрачные роли и доступы**: пользователь всегда понимает, что ему доступно.
3. **Пошаговый UX**: от входа к участию в событии за минимальное число действий.
4. **Отказоустойчивость UI**: ошибки не должны приводить к полной неработоспособности экрана.
5. **Детализированная структура интерфейса**: на каждом уровне (`Page` → `Screen` → `Multiwidget` → `Widget`) определены границы и ответственность.
6. **Максимальный упор на производительность.**

### 1.3 Иерархия UI (обязательная терминология)
Используется только следующая терминология:
- **Application**
- **Pages**
- **Screens**
- **Multiwidgets**
- **Widgets**

Базовая иерархия:
`Application -> Pages -> Screens -> Multiwidgets -> Widgets`

## 2. Полномочия пользователей и роли

### 2.1 Роли в приложении
1. **guest** (неавторизованный пользователь, только Login).
2. **user** (базовый авторизованный пользователь).
3. **manager** (пользователь-организатор/менеджер внутри конкретного события).
4. **admin** (системный администратор).

### 2.2 Полномочия по ролям

#### guest
- Просмотр только `LoginPage`.
- Авторизация через провайдеры OAuth.

#### user
- Просмотр ленты событий.
- Фильтрация и поиск.
- Просмотр деталей событий.
- Подача/отмена заявки на участие.
- Участие в голосовании (если событие допускает).
- Общение в чате события при наличии доступа.
- Просмотр и редактирование собственного профиля.
- Создание события.

#### manager (в рамках своего события)
- Все полномочия `user`.
- Редактирование параметров своего события.
- Управление участниками и заявками.
- Выбор и фиксация финального слота времени.
- Управление расходами события.
- Блокировка слотов для предотвращения голосования за них.

#### admin
- Все полномочия `user`.
- Доступ к `AdminPage`.
- Модерация пользователей и событий.
- Блокировка/разблокировка пользователей.
- Доступ к журналам действий.
- Восстановление/отмена событий по регламенту.

## 3. Ограничения стандартных ролей

### 3.1 Ограничения guest
- Нельзя видеть содержимое `EventListPage`, `ProfilePage`, `AdminPage`.
- Нельзя взаимодействовать с событиями, чатами, расходами.

### 3.2 Ограничения user (без premium)
- Лимит создаваемых событий: до 4 за 30 дней.
- Горизонт планирования: не далее 30 дней от текущей даты.
- Не более 3 тегов на событие.
- Стандартная приоритизация в выдаче ленты.

### 3.3 Ограничения manager
- Управление только своими событиями (или событиями, где роль manager явно назначена).
- Невозможность ограничить права создателя события

### 3.4 Ограничения admin
- Может архивировать, восстанавливать отмененные события, блокировать/разблокировать пользователей независимо от уровня премиум, читать обращения пользователей. Повышать и понижать рейтинг события независимо от уровня премиум.

## 5. Технологии и принципы выбора решений

### 5.1 Клиент
- Flutter (Web + Mobile).
- Управление состоянием: `bloc`.
- Маршрутизация: `auto_route`.
- Локализация: `l10n` (`ru`, `en` на первом этапе).

### 5.2 Авторизация
Только OAuth-провайдеры:
- Google
- Apple
- Twitter

**Причина**: отказ от хранения паролей и реализации password-based auth в MVP из-за высокой стоимости безопасной реализации и сопровождения.

### 5.3 Карты
- Используются **ссылки на сторонние карты** (Google/Yandex/другие внешние URL).
- Встраиваемые карты не используются в MVP.

**Причина**: встроенные map SDK и embedded-карты увеличивают стоимость и сложность, что избыточно для MVP.

### 5.4 Изображения
- Используются внешние ссылки на изображения.
- Хранение изображений на сервере в MVP не реализуется.

**Причина**: хранение и CDN требуют дополнительных затрат по объему, безопасности и поддержке.

### 5.5 Хранение клиентских данных
- Локальные настройки/кэши: `SharedPreferences` (web: browser storage abstraction).
- Серверные данные считаются источником истины.

#### D) Event card specification
`event_card` — карточка мероприятия, отображается списком в  `EventListPage`.

Параметры:
- минимальная ширина карточки - 500px.
- Скругление: 16 px.
- Внутренний padding: 12 px.
- соотношение ширины к высоте: 3х1. Максимальная ширина - 600px.
- цвет рамки такой же как у соответствующего фильтра (Карточка события должна иметь такую же рамку как и фильтры (мои, участвую, заявки), чтобы можно было сопоставить события в общей ленте.

Структура event_card :
Слева на право:
1. Фотография мероприятия/стандартная фотоплашка.
   В нижней части фото, поверх фото список тегов события (максимум 3)
2. Зона деталей события
   - Элементы сверху вниз:
     1. Заголовок(выравнивание слева) + цена мероприятия(если есть. Выравнивание справа).
     2. Краткое описание
     3. Дата/время
     4. Локация (кликабельная ссылка на карту) и текстовый адрес(если есть)
     5. Максимальное количество участников (если указано)
     Следующие элементы отличаются в зависимости от типа события

Типы событий:
1. `FixedEvent`
   - Имеет фиксированную дату и время проведения.
   - Основные действия: вступление/выход из участников, просмотр деталей.
2. `VotingEvent`
   - Имеет топ-3 слота, клик на которые создает/удаляет заявку на слот
   - Имеет раскрываемую зону голосования.
   - В раскрытии:
     - toggle-переключатель `table/list` (1:1). Подсвечивается выбранный вариант
     - Если выбрана таблица:
        - Таблица 7колонок(дни недели) х 12 строк(часы с шагом 1 час)
        - Заголовки колонок имеют формат Пн 18 янв.
        - Если диапазон голосования больше одной недели, таблица имеет пагинацию и переключатели по бокам.
        - Заголовки строк - 11:00.
        - Ниже таблицы элемент переключения между диапазонами 12:00-23:00(по умолчанию) и 00:00-11:00.
        - Заголовки строки колонок кликабельные. Клик выделяет/снимает все заявки на видимые пользователю сллоты.
    - Если выбран список:
        - Ширина карточки делится пополам 1:1.
        - Левая зона - список дат (диапазон голосования)
        - правая зона(при выборе даты) - список часов от 00:00 до 23:00.
        - Клик по выбранному часу создает/снимает заявку на данный слот.
    Дизайн слотов:
        - заблокированные. Серый цвет. Некликабельные.
        - Доступные для выбора - белые/черные в зависимости от темы.
        - Слоты за которые голосавали - слабо зеленые.
        - Топ-3 слота 0 ярко зеленые.
        - На слотах имеющих заявки на участие отображается количество заявок за этот слот.

- Правый сайдбар: `drawer`.
- Карточки: 100% ширины контента.

## 8. Screens вне самостоятельных Pages

### 8.1 EventCreateScreen
- Полнофункциональный Screen, открывается как modal/fullscreen flow из `EventListPage`.
- Поля сверху вниз:
  1. Название*
  2. Описание
  3. Ссылка на карту* + адрес + выбор города
  4. Теги (max 3)
  5. Публичность* (по умолчанию: публичное)
  6. Тип события* (`голосование` / `фиксированное`)
  7. Дата старта (только для фиксированных)
  8. Период голосования (только для голосования) (Строгое ограничение: дата окончания не меньше дата старта+1)
  9. Лимит участников (опционально)
  10. Цена (опционально)
  11. Ссылка на фото  (опционально)
  12. Для голосования - таблица где можно вычеркнуть слоты из голосования.
  13. Кнопка создания

### 8.3 VotingManageZone
- Доступно для менеджера события на `event_card`.
- Состав сверху вниз:
  1. Фиксация финального слота (вручную / автоматически).
  2. Рассылка уведомлений участникам.
  3. Отмена мероприятия (до начала). Тогда мероприятие переносится в архив с пометкой "отменено".
  4. Запрос на восстановление отмененного мероприятия к админу (за взнос). Работает только в случае если дата и время события еще не наступили.

## 9. Структура проекта

## 9.1 Архитектурные принципы
- Чистая архитектура: `data`, `domain`, `presentation`.
- Принципы: `DRY` + `KISS`.
- `core` и `l10n` — общепроектные инфраструктурные зоны, не относящиеся напрямую к трем слоям чистой архитектуры, но обязательные в архитектуре приложения.

## 9.2 Корневая структура
- `lib/core` — DI, router, общие утилиты и общая инфраструктура.
- `lib/l10n` — локализация и generated localization файлы.
- `lib/data` — источники данных, модели, репозитории-реализации.
- `lib/domain` — сущности, use cases, контракты репозиториев.
- `lib/presentation` — Pages/Screens/UI-state.

## 9.3 Структура feature в presentation
Для каждой Page/feature использовать единый шаблон:
- `page.dart` — контейнер Page.
- `screens/` — Screen-уровень данной Page.
- `widgets/` — Multiwidgets/устойчивые композиции.
- `models/` — presentation-модели.
- `bloc/` — state, events, bloc.

Пример:
`presentation/event_list/`
- `event_list_page.dart`
- `screens/`
- `widgets/`
- `models/`
- `bloc/`

## 9.4 Presentation core
В `presentation/core/` выносятся элементы, используемые несколькими Pages:
- language switcher,
- theme controls,
- shared layout helpers,
- повторно используемые UI-композиции.

## 10. Общие рекомендации по стилю и UX

### 10.1 Отступы и размеры
- Базовый шаг: 4 px.
- Основные отступы: 8 / 12 / 16 / 24 px.
- Скругления: 8 / 12 / 16 px.

### 10.2 Цвета и фон
- Светлая тема: светлый фон без темных пятен.
- Темная тема: черно-синий градиент фона.
- Использовать токены темы, не вводить случайные цвета.

### 10.3 Визуальная чистота
- Не допускать визуального шума.
- Избегать перегруженных панелей и дублирующих действий.
- Структурировать интерфейс через четкие зоны и разделители.

### 10.4 Технические стандарты
- Использовать `bloc` для сложного состояния.
- Использовать `auto_route` для навигации.
- Все пользовательские строки — через `l10n`.
- Не выносить отдельные файлы под элементарные одноразовые widgets.
- Выносить в `widgets/` только устойчивые переиспользуемые композиции.
- Хранение глобальных настроек пользователя в базе на бэкенде. Хранение состояния страницы в local_storage/shared_preferences браузера и восстанавливать при перезагрузке страницы.

## 11. Критерии качества реализации

Изменение считается корректным, если:
1. Не нарушена иерархия `Application -> Pages -> Screens -> Multiwidgets -> Widgets`.
2. Структура каждой Page описана и реализована последовательно (сверху вниз, слева направо).
3. Размеры/пропорции/позиционирование элементов соблюдены.
4. Ролевые права и ограничения соблюдены.
5. Технологические ограничения MVP соблюдены (OAuth-only, внешние карты, внешние изображения).
6. Проектная структура соответствует чистой архитектуре и правилам разделения слоев.
7. Терминология используемая в интерфейсе не использует хардкод, а полностью построена на ключах локализации arb.

### Новые вводные пользователя
- На карточках не показывать текстовый тип `голосование/фиксированное`: это лишняя информация.
- На карточке справа вместо текста `Введите сообщение` нужен только icon-only action чата.
- Кнопка участников должна быть icon-only (иконка пользователей).
- Для `fixed` карточек блок участников показывать просто как емкость `X/Y`.
- Для `voting` карточек показывать две строки:
  - `Максимальное количество участников: N`
  - `Претендентов на участие: M`
- Подтвердить, что интерфейс должен быть без хардкода строк (только через l10n).

- core должен содержать только действительно переиспользуемые виджеты, относящиеся к нескольким page/feature.
- Виджет, используемый только в одном ..._page, должен лежать в ..._page/widgets, а не в core/widgets.
- В каждом экране(page/features) должен быть только один, основной page (event_list_page), а вложенные элементы уровня screen должны лежать в screens.
